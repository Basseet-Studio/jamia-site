# Data Model: Veeramangalam Juma Masjid Household Finance Dashboard

**Branch**: `001-household-finance-dashboard` | **Date**: 2026-06-09
**Source**: `spec.md` Key Entities (L358-368) + `main spec.md` §1 Data Layer

Defines every persisted entity, fields, types, validation rules, state transitions, and derived values. The Firestore shape is locked in `firestore.rules` and implemented in `src/lib/services/`.

---

## 1. Admin

**Path**: `admins/{googleUID}`

| Field | Type | Required | Notes |
|---|---|---|---|
| `email` | string | yes | human reference only; not used for matching |
| `displayName` | string | yes | shown in nav + provenance |
| `role` | `"owner" \| "admin"` | yes | identical in-app permissions in v1; stored for future use |
| `addedAt` | Timestamp | yes | server timestamp at doc creation |

**Validation** (Zod schema in `src/lib/schemas/admin.ts`):
- `email`: valid email
- `displayName`: 1-100 chars
- `role`: enum

**Lifecycle**:
- Created manually in Firebase Console by the owner (no in-app UI per FR-005 + "Not in v1")
- Never updated by the app
- Never deleted by the app

**Invariant**: a user is authorised iff `admins/{request.auth.uid}` exists. Enforced both in `firestore.rules` and in `AuthGuard`.

## 2. Setting (singleton)

**Path**: `settings/global`

| Field | Type | Required | Notes |
|---|---|---|---|
| `defaultContributionTarget` | number | yes | starting target copied to new families |
| `openingBalance` | number | yes | seed value for money on hand; 0 if never set (FR-039a) |
| `currency` | string | yes | display label only, no conversion (FR-046) |

**Validation**:
- `defaultContributionTarget`: integer ≥ 0
- `openingBalance`: number (can be negative — reflects prior deficit)
- `currency`: 1-8 chars

**Lifecycle**:
- Read on every screen (real-time listener via `subscribeSettings`)
- Edited only from `/settings` with explicit warning on `openingBalance` change (FR-045)
- Singleton — no list, no delete

**Invariant**: the doc MUST exist before any other write. A deploy step (`scripts/seed-settings.ts`) seeds it via `pnpm seed:settings`.

## 3. Household

**Path**: `households/{householdId}`

| Field | Type | Required | Notes |
|---|---|---|---|
| `name` | string | yes | display name; unique within the org (case-insensitive) |
| `createdAt` | Timestamp | yes | server timestamp |
| `createdBy` | string (UID) | yes | admin who created it |

**Validation**:
- `name`: 1-80 chars, trimmed

**Lifecycle**:
- Created via "Add Household" dialog
- Hard deleted via "Delete Household" with name-retype confirmation (FR-014, FR-015). Cascade removes all families + all payments under them, atomically

**Invariant**: deletion is atomic — single batched write across the household doc, all family docs, and all payment sub-collection docs.

## 4. Family

**Path**: `households/{householdId}/families/{familyId}`

| Field | Type | Required | Notes |
|---|---|---|---|
| `name` | string | yes | display name within the household |
| `contributionTarget` | number | yes | monthly target; copied from global default at creation |
| `createdAt` | Timestamp | yes | server timestamp |
| `createdBy` | string (UID) | yes | admin who created it |
| `active` | boolean | yes | `true` by default |
| `deletedAt` | Timestamp \| null | yes | `null` until soft-deleted |
| `deletedBy` | string (UID) \| null | yes | `null` until soft-deleted |

**Validation**:
- `name`: 1-80 chars
- `contributionTarget`: integer ≥ 0
- `active`: boolean
- `deletedAt` / `deletedBy`: present iff `active === false`

**Lifecycle**:
- Created via "Add Family" dialog on Household Detail. `contributionTarget` defaults to `settings.global.defaultContributionTarget`; admin can override at creation.
- Soft-deleted: `active → false`, `deletedAt` set, `deletedBy` set. The document is NEVER removed from Firestore. Payments sub-collection is NEVER touched.
- No restore in v1.
- `familyId` is auto-generated by Firestore. NEVER reused.

**Invariant (FR-012)**: family IDs are reserved forever. Service layer uses `addDoc` and never accepts a chosen ID.

## 5. Payment

**Path**: `households/{householdId}/families/{familyId}/payments/{paymentId}`

| Field | Type | Required | Notes |
|---|---|---|---|
| `amount` | number | yes | positive decimal |
| `date` | Timestamp | yes | the date the admin assigned (default: today) |
| `month` | string | yes | `"YYYY-MM"` derived from `date` at write time |
| `note` | string \| null | yes | optional; up to 280 chars |
| `recordedAt` | Timestamp | yes | server timestamp of write |
| `recordedBy` | string (UID) | yes | admin who recorded it |

**Validation**:
- `amount`: number > 0
- `date`: any valid Timestamp
- `month`: regex `^\d{4}-(0[1-9]|1[0-2])$`, must match `format(date, "yyyy-MM")` of `date` (enforced by service layer)
- `note`: string up to 280 chars, may be empty

**Lifecycle**:
- Created via "Record Payment" dialog. Multiple payments per family per month are allowed (FR-018).
- Deleted via "Delete" on the payment row. Permanent. Confirmation required (FR-019).
- No edit action in v1 (FR-020, US-9).

**Invariant**: `month` is computed at write time from `date`, never edited separately. Soft-deleting the parent family does NOT cascade to payments — they remain (FR-013).

**Derived value (not stored)**:
- **FamilyMonthlyStatus**: `totalPaid = sum(payments.amount where month == M)`. Status = `Unpaid` if `0`, `Partial` if `0 < totalPaid < target`, `Met` if `totalPaid == target`, `Over` if `totalPaid > target`. Considers only active families (FR-023).

## 6. Expense

**Path**: `expenses/{expenseId}`

| Field | Type | Required | Notes |
|---|---|---|---|
| `name` | string | yes | display name |
| `amount` | number | yes | positive decimal |
| `date` | Timestamp | yes | date assigned by admin |
| `month` | string | yes | `"YYYY-MM"` derived from `date` at write time |
| `note` | string \| null | yes | optional, up to 280 chars |
| `isRecurring` | boolean | yes | `true` if created from a template |
| `recurringId` | string \| null | yes | source template ID, or null |
| `withdrawn` | boolean | yes | `false` initially |
| `withdrawnAt` | Timestamp \| null | yes | `null` until withdrawn |
| `withdrawnBy` | string (UID) \| null | yes | `null` until withdrawn |
| `addedAt` | Timestamp | yes | server timestamp |
| `addedBy` | string (UID) | yes | admin who added it |

**Validation**:
- `amount`: number > 0
- `date`: valid Timestamp
- `month`: regex `^\d{4}-(0[1-9]|1[0-2])$`
- `isRecurring`: boolean
- `recurringId`: present iff `isRecurring === true`
- `withdrawn`: boolean
- `withdrawnAt` / `withdrawnBy`: present iff `withdrawn === true`

**Lifecycle states**:
```
[NotWithdrawn] --admin clicks Withdraw + confirms--> [Withdrawn]
[NotWithdrawn] --admin clicks Delete + confirms--> [deleted]
[Withdrawn]    --admin clicks Delete + confirms--> [deleted]
```

State transitions enforced by service layer (`expenses.ts`):
- `withdraw(expenseId)` only when `withdrawn === false`. Sets `withdrawn: true`, `withdrawnAt: serverTimestamp()`, `withdrawnBy: auth.uid`.
- `delete(expenseId)` allowed in both states. Effect on money on hand: if `withdrawn === true`, money on hand increases by `amount`; otherwise unchanged.
- No undo in v1 (FR-031).

**Derived values (not stored)**:
- **MonthlyExpenseSummary** (per month M): `totalAdded = sum(expenses.amount where month == M)`, `totalWithdrawn = sum(expenses.amount where month == M AND withdrawn)`, `totalPending = totalAdded - totalWithdrawn`.
- **AllTimeExpenseSummary**: `allTimeAdded = sum(all expenses.amount)`, `allTimeWithdrawn = sum(all expenses.amount where withdrawn)`.

## 7. RecurringExpenseTemplate

**Path**: `recurringExpenses/{recurringId}`

| Field | Type | Required | Notes |
|---|---|---|---|
| `name` | string | yes | display name |
| `amount` | number | yes | positive decimal |
| `description` | string \| null | yes | optional, up to 280 chars |
| `active` | boolean | yes | `true` = visible in active list |
| `createdAt` | Timestamp | yes | server timestamp |
| `createdBy` | string (UID) | yes | admin who created it |

**Validation**:
- `name`: 1-80 chars
- `amount`: number > 0
- `description`: string up to 280 chars, may be empty
- `active`: boolean

**Lifecycle**:
- Created via "Add Template" dialog on Recurring screen.
- Editable (name, amount, description) without affecting already-created expense rows (FR-037).
- Archived via "Archive" action — sets `active: false`. Previously generated expenses are preserved (FR-038).
- Templates NEVER auto-create expense rows (FR-034, US-7, assumption).

**Derived value (per template, per current month)**:
- `RecurringTemplateMonthStatus`:
  - "NotAdded" if no `expenses` doc has `recurringId == template.id AND month == currentMonth`
  - "PendingWithdrawal" if such a doc exists and `withdrawn === false`
  - "Withdrawn" if such a doc exists and `withdrawn === true`
- Computed client-side by joining `recurringExpenses` with the `expenses` query `where("recurringId", "in", activeTemplateIds).where("month", "==", currentMonth)`.

## 8. MoneyOnHand (derived, NOT stored)

**Path**: N/A — computed live from the three source collections.

```
moneyOnHand =
  settings.global.openingBalance
  + Σ payments.amount (ALL payments, active AND inactive, all time)
  - Σ expenses.amount where expenses.withdrawn === true (all time)
```

**Reactive computation**:
- `subscribeMoneyOnHand(callback)` subscribes to:
  - `settings/global` (1 doc)
  - ALL payment docs (collection-group query on `payments`)
  - ALL expense docs where `withdrawn === true`
- All three subscriptions are live (`onSnapshot`); the formula is recomputed on every change.
- Returns `{ value: number, currency: string, asOf: Timestamp }`.

**Invariants (SC-009)**:
- Deleting a payment → money on hand decreases by exactly `payment.amount`.
- Withdrawing an expense → money on hand decreases by exactly `expense.amount`.
- Deleting a withdrawn expense → money on hand increases by exactly `expense.amount`.
- Deleting a not-withdrawn expense → money on hand unchanged.
- Changing `openingBalance` by `Δ` → money on hand changes by exactly `Δ`.

These are tested in `tests/unit/services/moneyOnHand.test.ts` and the E2E specs.

## 9. HouseholdMonthlySummary (derived, NOT stored)

**Path**: N/A — computed live per household per month.

```
totalFamilies      = count(families where active)
familiesPaidFull   = count(families where active AND status in {Met, Over})
familiesPartial    = count(families where active AND status == Partial)
familiesUnpaid     = count(families where active AND status == Unpaid)
totalCollected     = sum(payments.amount where month == M across all families in household, active AND inactive)
totalTarget        = sum(families.contributionTarget where active)
collectionRate     = totalCollected / totalTarget  (as percentage; null if totalTarget == 0)
```

**Reactive**: `subscribeHouseholdMonthlySummary(householdId, month, callback)`. The dashboard pre-computes this for all households in a single pass.

## 10. Dashboard summary (derived, NOT stored)

Computed for the current month. Surfaces:
- Money on hand (all time, per §8)
- This month's collections vs target (sum across all households)
- This month's expenses added vs withdrawn
- Number of households
- Quick stat: families fully paid this month vs total active families

The dashboard subscribes to the three live sources (settings, payments, expenses) and recomputes locally.

## 11. Indexes

Composite indexes required (declared in `firestore.indexes.json` at repo root):

1. `payments` — collection group
   - `month` ASC, `recordedAt` DESC
   - (used for family history filtered by month, sorted newest first)

2. `expenses`
   - `month` ASC, `withdrawn` ASC, `date` DESC
   - (used for monthly expense summary + table)

3. `expenses`
   - `recurringId` ASC, `month` ASC
   - (used for recurring template "added this month?" check)

4. `households/{householdId}/families`
   - `active` ASC, `name` ASC
   - (used for household detail family table)

The `payments` collection-group query is the only one that needs the v1 special handling: it spans all households, all families, so it MUST be a collection group query to compute money on hand.

## 12. State transitions summary

| Entity | States | Transitions |
|---|---|---|
| Family | `active` ↔ `inactive` (soft) | Admin: soft delete (one-way in v1, no restore) |
| Expense | `notWithdrawn` → `Withdrawn` | Admin: Withdraw (confirmation) |
| Expense | any → `deleted` | Admin: Delete (confirmation) — affects money on hand only if was `Withdrawn` |
| RecurringTemplate (per month) | `NotAdded` → `PendingWithdrawal` → `Withdrawn` | Admin: Add for This Month, then Withdraw on Expenses screen |
| RecurringTemplate (archived) | `active` → `archived` | Admin: Archive |

All other entities (Admin, Setting, Household, Payment) are create + read + delete (household is hard delete cascade). Payment has no edit in v1.
